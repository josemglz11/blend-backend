# BLEND Backend

Production-ready Node.js backend for the BLEND custom protein platform.

## Architecture

```
src/
  index.js                    Server entry point, middleware, route mounting
  seed.js                     Catalog seed data (bases, flavors, add-ins)
  services/
    RecommendationEngine.js   Rules-table quiz recommendation engine (versioned)
    BlendSpecService.js       BlendSpec creation, pricing, macros, cloning
    SubscriptionService.js    Full subscription state machine (7 states)
    OrderService.js           Order creation, pick-pack doc generation
  routes/
    index.js                  All API routes + Stripe webhook handler
  workers/
    subscriptionWorker.js     Cron workers: ship, payment retry, auto-resume, cancel
prisma/
  schema.prisma               Full data model
```

## Data Model Overview

The **BlendSpec** is the canonical object. It represents one customer's
formula at a point in time (base, flavor, dose, add-ins). Everything else
references it.

A **Subscription** holds a `blendSpecId` (the live formula) and optionally
a `pendingBlendSpecId` (a modification queued for the next cycle).
When the ship worker processes a cycle, it commits the pending spec
and advances dates.

Every subscription state transition is recorded as a **SubscriptionEvent**,
giving you a full audit trail with prior state, new state, reason, and
whether it was user-driven.

An **Order** is created per ship cycle and carries a complete snapshot of the
**pick-pack document** your co-packer needs to produce and ship one bag.
This doc is immutable once created.

## Subscription State Machine

```
ACTIVE
  ACTIVE_PENDING_MODIFICATION   (user changed their next blend)
  PAUSED                        (user paused, optional resume date)
  SKIP_NEXT                     (one cycle skipped, auto-resumes)
  PAYMENT_FAILED                (Stripe failed, retry up to 4 times)
  CANCEL_PENDING                (cancel requested, takes effect at period end)
  CANCELED                      (terminal, BlendSpec preserved for resurrection)
```

## Setup

### 1. Prerequisites

- Node.js 20+
- PostgreSQL database (local or hosted, e.g. Supabase, Railway, Neon)
- Stripe account
- Resend account (for transactional email)

### 2. Install

```bash
cd blend-backend
npm install
```

### 3. Configure

```bash
cp .env.example .env
# Fill in DATABASE_URL, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, etc.
```

### 4. Database

```bash
npm run db:generate   # Generate Prisma client
npm run db:migrate    # Run migrations
npm run db:seed       # Seed catalog (bases, flavors, add-ins)
```

### 5. Run

```bash
npm run dev    # Development (auto-restarts on file change)
npm start      # Production
```

## Authentication

The routes currently use a simple `x-customer-id` header for auth.
In production, replace the `requireAuth` middleware in `src/routes/index.js`
with your preferred auth system: JWT, Clerk, Auth0, Supabase Auth, etc.

## Co-packer Integration

The co-packer pick-pack document is generated by `OrderService._buildPickPackDoc()`
and stored as JSON on the Order record. It contains everything your co-packer
needs: base SKU, flavor SKU, fill weight, add-in sachets, label data, shipping
address, QC checklist.

To connect to your co-packer:
1. Set `COPACKER_WEBHOOK_URL` and `COPACKER_API_KEY` in `.env`.
2. The `sendToCopackerWebhook()` function in `src/workers/subscriptionWorker.js`
   will POST each order's pick-pack doc to that URL on every ship cycle.
3. Update the SKU codes in `OrderService._buildPickPackDoc()` to match your
   co-packer's actual inventory codes.

## Adding Rule Set Versions

To update the recommendation logic without breaking existing customers:
1. Copy `RULE_SET_V1` in `RecommendationEngine.js` to `RULE_SET_V2`.
2. Make your changes.
3. Update `CURRENT_RULE_SET_VERSION` to `'2.0.0'`.
4. The engine stores `ruleSetVersion` on every BlendSpec, so you can always
   replay old quiz answers against the rule set that was live when they were taken.

## Workers

Four cron workers run automatically when the server starts:

| Worker | Schedule | Purpose |
|--------|----------|---------|
| ShipWorker | Daily 6am UTC | Charge, create order, generate pick-pack doc |
| PaymentRetryWorker | Mon + Thu 8am UTC | Retry failed payments, cancel after 4 failures |
| AutoResumeWorker | Every hour | Resume paused subs past their resume date |
| CancellationFinalizer | Daily 7am UTC | Finalize CANCEL_PENDING at period end |
